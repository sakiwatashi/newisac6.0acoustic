## 2. 若只做一件事：D2 設計敏感度（規格）

### 2.1 目標

在**同一 30 個目標位置**（與正式 `v2_d2v2_formal` 同 seed／同序列）上，證明：

> 五點是「最低三點需求 + 對稱冗餘」的工程折衷，並用數據說明三點較不穩、七點收益有限（若結果如此）。

**不覆寫** `v2_d2v2_formal`；新結果寫入新目錄，論文稱「設計敏感度分析」。

### 2.2 最小實驗矩陣（推薦：5 個新條件 + 重用正式 5 點）

| ID | 視點 y (m) | 基線跨度 | 備註 |
|----|------------|----------|------|
| **K5_B30** | −0.15…+0.15 五點 | 0.30 | **重用正式** closed 定位／停止指標 |
| K3_B30 | −0.15, 0, +0.15 | 0.30 | 最低對稱 |
| K7_B30 | linspace(−0.15,0.15,7) | 0.30 | 加點 |
| K5_B15 | linspace(−0.075,0.075,5) | 0.15 | 窄基線 |
| K5_B45 | linspace(−0.225,0.225,5)* | 0.45 | 寬基線；\*需 IK／包絡閘門，過不了則記錄 FAIL |

\*寬基線若 IK 或姿態不過 → **如實記不可達**，本身即設計結論。

掃描中心：**名義工作區 y=0、前向 sensor x=0.60**（與正式一致）；**不用該回合目標真值**選中心。

### 2.3 固定項

- 同 seed、同 30 目標（closed 臂即可做定位敏感度；若要停止 RMSE 可只跑 closed）  
- 同 bar 校正、同 trilateration（Gauss-Newton 圓）  
- 同 standoff／控制（若跑完整接近）  
- 同 IK／姿態判準  

### 2.4 必報指標

| 指標 | 用途 |
|------|------|
| \(r_x, r_y\), RMSE\(_x\), RMSE\(_y\) | 定位 |
| 2D 停止 RMSE（若跑接近） | 任務 |
| 定位失敗／n_used 視點 | 穩健 |
| IK／姿態違規 | 可達性 |
| 每回合量測次數、牆鐘或步數 | **成本** |

### 2.5 預註冊裁決（建議先寫進 runner header）

不強求「五點最優」，只預註冊**報告義務**：

1. 三點 RMSE_y 是否 ≥ 五點（預期：較差或失敗率高）。  
2. 七點相對五點的 RMSE_y 改善是否 < 某閾（例如 10% 相對）且成本增加。  
3. 0.15 vs 0.30 vs 0.45 之 RMSE_y 與違規率是否呈折衷而非單調。  

口試話術：**工程折衷 + 敏感度數據**，不宣稱全域最佳。

### 2.6 實作落點（repo）

| 項目 | 路徑構想 |
|------|----------|
| Runner | `scripts/d2v2_design_sensitivity_runner.py`（fork formal，CLI：`--n-vantages` `--baseline-m`） |
| Shell | `runtime/run_v2_d2_sensitivity.sh` |
| 輸出 | `runtime/outputs/v2_d2_sensitivity_k{3,5,7}_b{15,30,45}/` |
| 分析 | `scripts/analyze_d2_sensitivity.py` → 總表 JSON/CSV |
| 報告 | `docs/plan_v2/reports/D2_DESIGN_SENSITIVITY_REPORT.md` |
| 論文 | 5.4 增「設計敏感度」一小節；正式五點表不動 |

### 2.7 工作量粗估

| 模式 | 量 | 粗估 |
|------|-----|------|
| 僅定位（五視點掃描、不完整接近） | 4 新條件 × 30 目標 | 約數小時 GPU（視 settle） |
| 完整 closed 接近 | 同上 × 接近步 | 明顯更長 |

**建議先定位-only** 回答「為什麼五點」；停止 RMSE 可選。
